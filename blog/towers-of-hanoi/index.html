The problem

Imagine we have three towers and want to move a pile of disks stacked from top to bottom smallest to largest, from one tower to another. The catch is that:

  • We can only move one disk at a time.
  • A larger disk can not cover a smaller disk.

We want to represent our solution in code by outputting an array of each of the moves that we make.

An algorithm to solve it

Moving a pile of just one or two diskes is straight forward. However, We observe that moving 3 or more disks requires a lot of moves and is not a random affair.

[("A","B"),("A","C"),("B","C"),("A","B"),("C","A"),("C","B"),("A","B")]

And 4 disks looks even more involved:

[("A","C"),("A","B"),("C","B"),("A","C"),("B","A"),("B","C"),("A","C"),("A","B"),("C","B"),("C","A"),("B","A"),("C","B"),("A","C"),("A","B"),("C","B")]

Yet there is a pattern, and it’s recursive. We:

  1. Move n - 1 disks from a to c
  2. Move 1 disk for a to b
  3. Move n - 1 disks from c to b

Starting to code

By using types we can make our code a lot clearer, as if it were documentation; the word Tower adds semantic meaning.

type Tower = String
type Move = (Tower, Tower)

And, having a type signature makes it very clear what data we take in and produce and is generally very helpful.

hanoi :: Integer -> Tower -> Tower -> Tower -> [Move]

Base case/s

For recursion we always need base cases to end the recursion. And they will return values.

hanoi 0 _ _ _ =  []
hanoi 1 start end _ =  [(start, end)]

Recursive case

hanoi n start end temp =
  let nMinusOne = (n - 1)
    in
    hanoi nMinusOne start temp end ++
    hanoi 1 start end temp ++
    hanoi nMinusOne temp end start

Let’s look at how the recursion is actually working

THis is how we might call it:

hanoi 3 "a" "b" "c"

  • We don’t return any values. We do set nMinusOne to 2.

    hanoi 2 “a” “c” “b”

  • We have changed the order of our arguments/ towers.

  • We set nMinus One to 1. Now we can start producing values, which will be:

[(“a”,“b”),(“a”,“c”),(“b”,“c”),(“a”,“b”)]

  1. hanoi 1 start temp end produces [(“a”, “b”)]
  2. hanoi 1 start end temp produces [(“a”, “c”)]
  3. hanoi 1 temp end start produces [(“b”, “c”)]
  4. [(“a”, “b”)] is added now. I am not sure why that only happens now, and not in the next round.

we recur again with 2. I think that is because nMinusOne is lexically scoped at the start. The arguments now are: “c”“b”“a”. This time we produce one less value, however.

  1. hanoi 1 start temp end produces [(“a”, “b”)]
  2. hanoi 1 start end temp produces [(“c”, “b”)]
  3. hanoi 1 temp end start produces [(“a”, “b”)]
